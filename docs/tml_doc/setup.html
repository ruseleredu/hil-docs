    Description

    This is a documentation of the Typhoon Modeling Language.

    The project is under development. For additional resources (books, initial notes and analysis) check tml-docs repo

    Note

    Documentation is keyboard-friendly. Use arrows to turn pages or s for full text search. You can also use mouse and buttons on the top of the page.

    You can also change the theme, or print to pdf (upper right) if you prefer to read the docs in pdf format.

    Setting up TML for development

    Make sure you have currently supported Python version installed. You can use tools like asdf for multiple version management.

    Create Python virtual environment:

    python -m venv venv

    Activate the environment and install TML and dev dependencies:

    source venv/bin/activate      # for Linux/MacOS
venv\Scripts\activate.bat     # for Windows

pip install -e .[dev]

    Setting up THCC for development

    TML support is implemented on the tml-develop branch. TML compiler is automatically downloaded and installed alongside other compilers. To install TML compiler automatically, run update_envirnoment.sh script. Alternatively, only TML portion of _download_compilers.sh can be ran to save time.

    TML is disabled by default in THCC. TML_ENABLE feature flag in global_settings.py must be set to True to call TML compiler. Otherwise, Jinja components are used.

    If a component has both TML and Jinja implementations, TML is given precedence if enabled. Otherwise, Jinja is used.

    TML implementations of components must be placed inside t_sw/thcc/typhoon/lib_parts/components/tml directory for compiler to find them. Names of components must be the same as for Jinja implementations, but with .tml extension. TML-only components can exist, but will break model configuration if TML is later disabled.

    Development process

    The branch for TML development is main We shall tag each new release with x.y.z where x.y.z are written according to Semantic Versioning.

    All TML development goes with this simple process:

    1. Create issue and branch (draft merge request) forked of the main branch.
    2. Develop and comment on the issue if needed. Write each design decision we negotiated to the docs. If the issue is still unsolved or the issue discussion is relevant for the future work (i.e. todo or question admonitions) link to the issue where the discussion took place with the standard GitLab #x syntax.
    3. Milestones TML-x.y are used for planning next releases. Each issue planned for the next and the release after next should have milestone defined.
    4. There is an issue board which provides an overview of the current development.
    5. Write test for each new/changed feature
    6. Be sure that all the test pass. If a detailed review is needed assign a person for the code review and move the issue to W::REVIEW label/state. If review is not necessary merge to main, close the issue and delete the branch.

    Running tests

    Tests are located in the tests folder. To run use:

    pytest tests/

    Note

    We keep tests that are not passing in a separate folder tests/skipped. These tests are those that can't be fixed for a longer period of time.

    All tests in other folders should always pass.

    The tests operate by parsing the .tml files and comparing the result with the corresponding output file (.output, .c etc.). If these don't match the test fails.

    When there is no output file when a test is run a new output file is generated whose contents corresponds to the parsed .tml file.

    The easiest way to update output files when you change the .tml code is to just delete its corresponding output files. New files will be generated and you can compare the result using git diff to see if the difference is expected. If so, commit files to the git repo as they now form a new baseline.

    Docs installation and setup

    We are using mdbook for TML docs.

    For easy setup of reproducible environment for document development TML project uses Nix.

    To read the docs on your dev machine install Nix.

    Be sure to configure Nix flake feature by adding experimental-features = nix-command flakes to ~/.config/nix/nix.conf. See more here.

    To get a shell with all dependencies installed run:

    nix develop

    from docs folder of the project repo. After necessary dependencies are downloaded and setup you will get a shell with everything configured for working with mdbook.

    On this shell run:

    mdbook serve

    To serve documentation locally.

    The book will be available at http://localhost:3000/

    Instead of running nix develop you can install direnv which handles this automatically when you cd into docs folder.

    Tip

    If for some reason you can't use Nix there is an option to run docs through a docker image.

    Install docker and run script serve-docs.sh from the root of the project. Stop the instance by running docker stop tml-docs from another terminal.
