    IO

    IO variable representation

    TML represents IO interfaces to users via IO variables (described in detail in user documentation). IO variables are added to the symbol table, like user and model scope variables. Every symbol table entry is assigned a name, type and object. Symbol name and type are used for ordinary semantic analysis, applied to all expressions in TML. The object denotes special IO-related attributes needed in code generation. While name and type enable abstract interface over configurable IO, the object enables specialized logic code generation for a concrete instance of model and component configuration.

    IO variable types

    Currently, TML only supports IO variables that represent communication with FPGA solver. Two supported ways of communicating with FPGA solver are basic IO and DMA. The TML compiler decides concrete IO implementation depending on user settings, IO variable declaration and device capability. Declaring a DMA variable does not mean that the compiler will enforce DMA mechanism. DMA must be enabled in model settings and supported by the device in order to be used. If enabled and supported, all model variables declared as DMA variables will use DMA, otherwise basic IO is used as a fallback.

    For both FPGA IO variable types, the end address of the destination IO device must be manually specified. TML allows only expressions that can be statically evaluated to be used for addresses to prevent any possible ambiguities about resolution order. IO variable addresses are resolved when compiling components. If addresses need to be parameterized using hardware or model configuration, address values must be calculated inside component handlers and passed via model scope.

    Basic IO

    The main interface used for basic IO by TML are atomic HIL_InInt32/HIL_OutInt32 and HIL_InFloat/HIL_OutFloat functions. Atomic functions work on scalar types. Scalar writes consist of calling IO functions with value and address as parameters. Vector writes consist of calling a sequence of atomic operations, with the address being incremented by one for every next element. If broadcasting is applied, the same scalar value is written on all addresses belonging to vector IO variable address range.

    DMA

    HIL devices support scatter-gather DMA approach. This means that destination IO devices are viewed as non-contiguous (i.e. scattered) address ranges, while structures for reading and writing values from and to IO devices are contiguous address ranges (hence IO gathers values on read and scatters them on write). DMA has 12 read channels and 12 write channels. Every channel is a separate contiguous address space visible to TML. End device destination addresses are the same as in basic IO and the same manual mechanism is used for resolving them. Internal DMA structure and channel addresses are automatically resolved. Upon resolving internal DMA addresses, the DMA controller is configured to map internal and external addresses.

    DMA execution cycle

    Before explaining how code for DMA is generated, we must first understand how DMA works and what are the main rules for DMA controller configuration and DMA channel allocation. Lower ID CPU has higher priority than higher ID CPU (i.e. CPU0 has higher priority than CPU1). Faster execution rate has higher priority than a slower one. When allocating channels, used CPUs are first sorted, followed by sorting ERs per CPU. Read and write DMA channels are allocated separately.

    Detailed DMA execution steps are:

    1. During model initialization, for every DMA channel, DMA controller must be configured with external space address and size and matching internal address
    2. Before running simulation cycle, DMA read synchronize sequence must be performed to move all current input values from external address spaces to internal ones
    3. After running simulation cycle, DMA write synchronize sequence must be performed to move all current output values from internal address spaces to external ones

    DMA code generation

    Direct association of DMA variables and channels would make compilation significantly more complicated. DMA channels cannot be allocated before all TML components are analyzed, as the number of DMA variables is not known before type analysis (preceded by parsing and macro expansion). Such kind of interdependency presents a common problem in compilers: an example would be how linkers are used to combine separate portions of code. Thus, TML compiler adopts context-free approach. Every DMA variable has unique name, and channels are referenced by their deterministic names that consist of IO direction, CPU ID and execution rate. TML compiler outputs context-free DMA variable metadata for every component, and only after all components were analyzed, the compiler metadata is used for allocating DMA channels and DMA controller configuration sequences.

    Detailed steps performed on TML compiler side are:

    1. For every component, TML analyzes whether IO variable will be generated using basic IO or DMA (depending on model configuration, hardware configuration and IO variable declaration)
    2. For every DMA variable in component, TML compiler stores metadata about size, address and target CPU and ER
    3. IO operations are basic variable reads and writes, with only variable codegen name having DMA struct in prefix
    4. On component level, TML generates code that assumes constant channel name
    5. After all components were compiled, metadata is gathered and concrete DMA channels are assigned on model level. DMA channel metadata is exported for THCC

    Detailed steps performed on THCC side are:

    1. DMA channel and variable metadata is read
    2. IO channel structures are generated and global order of IO variables in internal address space is selected
    3. IO structs are assigned to address spaces with deterministic names that are allocated in linker script (using symbolic names wherever possible is good practice from code readability standpoint)
    4. DMA controller configuration sequences are generated for every DMA channel, manual external and automatic internal addresses are mapped
    5. DMA synchronization sequences are generated for every DMA channel
    6. Address spaces are generated in the linker script starting from lower to higher channel numbers. Channel addresses are cache-line size aligned as that is required by ARM cache-coherent communication interface
