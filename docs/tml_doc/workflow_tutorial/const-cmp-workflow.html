    Porting Singal Processing Components from Jinja Templates to TML - Workflow

    The following document describes the workflow of porting the existing Signal Processing components implemented using Jinja templates to Typhoon Modelling Language (TML). A simple constant component described with the src_constant.template file serves as an example.

    The first step assumes designing a model containing the targeted component using Typhoon Schematic Editor. As Figure 1 depicts, the model contains only a constant component connected to a probe component Probe1. After validating and compiling the model, one chooses an option to load the model to a Virtual HIL (Load Model to Virtual HIL in Figure 2) and examines it using HIL SCADA.

    Figure 3 shows the SCADA panel containing a gauge component inspecting the value of Probe1 output in real-time. The gauge shows the value 23 set by the TSE model shown in Figure 1.

    In a more general case, one can inspect the model's behavior in real-time using SCADA and use the inspected values as a reference while validating the future TML implementation of components.

    To implement the component in TML, one creates a .tml file with the same name as .template file (src_constant.tml in this case) and places it underneath t_sw/thcc/typhoon/lib_parts/components/tml folder. The following listing presents the trivial implementation of the component in the src_constant.tml file. It shows how the component copies a property's value to the output terminal.

    # Listing 1 - src_constant.tml contains implementation of the constant component.
fn output_fnc():
    t.out = p.value
end

    To validate the implementation, restart Typhoon Schematic Editor, open the previous model (Figure 1), recompile it and start the simulation in SCADA. If the output looks the same as Figure 2 shows, then the implementation seems correct.

    Further validation would require running standalone integration tests present in the t_sw/tests/20_standalone/ folder. One should position itself in the child folder corresponding to the component recently reimplemented in TML (436_constant_source in this case) and run the tests with the command like the following: pytest --vdev=HIL604 --vcfg=1. If the implementation is correct, there should be no failing tests, as Listing 2 shows.

    # Listing 2 - Running standalone integration tests for the constant component 
cd t_sw/tests/20_standalone/436_constant_source
pytest --vdev=HIL604 --vcfg=1
# Output:
# =========== 15 passed in 193.21s (0:03:13) ===========
