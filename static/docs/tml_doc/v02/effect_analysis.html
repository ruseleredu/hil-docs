Effect analysis

Effect system represents a way of complementing type system with information on what effects expression evaluation can cause. In TML, effects are used to determine which expressions can be statically evaluated, which functions are allowed to access terminals and which functions potentially do not terminate.

Supported effects are listed in the table below.

Effect Parameter Description
READ_GLOBAL Variable A global variable was read
WRITE_GLOBAL Variable A global variable was written
READ_OUTPUT Terminal,
flag that denotes access in feedthrough guarded context 		A terminal was read
WRITE_OUTPUT Terminal,
flag that denotes access in feedthrough guarded context 		A terminal was written to
READ_FEEDTHROUGH_OUTPUT Terminal,
flag that denotes access in feedthrough guarded context 		A feedthrough output terminal was read
WRITE_FEEDTHROUGH_OUTPUT Terminal,
flag that denotes access in feedthrough guarded context 		A feedthrough output terminal was written to
READ_INPUT Terminal,
flag that denotes access in feedthrough guarded context 		An input terminal was read
READ_FEEDTHROUGH_INPUT Terminal,
flag that denotes access in feedthrough guarded context 		An input terminal was written to
READ_TUNABLE_PROPERTY Tunable property A tunable property was read
GUARD_EXISTS Variable A variable was guarded using exists statement
GUARD_NOT_EXISTS Variable A variable was guarded using not exists statement
GUARD_FEEDTHROUGH Variable A variable was guarded using feedthrough statement
GUARD_NOT_FEEDTHROUGH Variable A variable was guarded using not feedthrough statement
NON_TERMINATING none A function cannot be proven to terminate
ACCESS_UNGUARDED_OPTIONAL Variable An optional variable was accessed outside exists statement
DERIVE_UNGUARDED_OPTIONAL Variable Type of an optional variable was derived outside exists statement

Effects can be stored at the expression level in the AST, ranging from the simplest forms (literals and variable references) to complex expressions. Current implementation stores effects at function level in a separate structure called effect scope.

Effects are not present in TML syntax and effect inference is performed automatically. Users are aware of effects only in the form of error messages such for actions such as forbidden terminal access.

Overview

Effect analysis is performed as an AST traversal, with separate passes for local and global scope.

Local scope analysis is performed before global scope analysis. Effects of functions are determined as the union of all effects caused directly by the function and by the callee functions.

Besides effect type and parameter variable, every effect holds guard context in which it was created. This enables detecting whether an effect is reachable in the context in which it is inherited. If it is reachable, it is inherited. Otherwise, it is not inherited and a warning is raised. Note that an effect might be reachable (and inherited) in one context, and unreachable (and discarded) in another.

Expressions in global scope must be statically evaluated, so only pure terminating functions can be called in global scope. A function is considered pure if it doesn't have any effects. A function is considered terminating if it is not recursive and does not have a while statement (either directly or in called code).

Local scope

Guard context

Guard context is defined either using guard blocks or inline guards. Guard context contains assumptions about variable properties that are valid in a certain portion of code. Properties of concern are presence of optional variables and feedthrough property value of dynamic feedthrough terminals. Guard context consists of 6 separate sets of assumptions, described in the table below. While behavior is same for both guard blocks and inline guards, sets are separate for easier readability of test and debug output.

Set Description
guarded_exists Set of terminals guarded using exists statement
guarded_not_exists Set of terminals guarded using not exists statement
guarded_feedthrough Set of terminals guarded using feedthrough statement
guarded_not_feedthrough Set of terminals guarded using not feedthrough statement
inline_guarded_exists Set of terminals guarded using inline guard operator ?
inline_guarded_not_exists Set of terminals guarded using inline guard operator ? in left elvis operator branch that are assumed not to be present in the right elvis operator branch

Total function effects

Function definition is represented as a subtree of AST, beginning with function_definition node. Functions are analyzed in the topological order. Functions that call no functions have no inherited effects and can be analyzed first. Dependencies are then analyzed before the dependent functions.

If a simple expression with a certain known effect is encountered during AST traversal, the effect is added to the effect scope belonging to the functions. For example, a statement such as t.in = 1 would add WRITE_TERMINAL<t.in> effect.

If a function call is encountered, all reachable callee effects are inherited. An effect is considered reachable if it was created in a guard context that is not opposite to the guard context in which the function call is placed. Terminal effects are mapped to their guarded counterparts, if necessary.

Guard context is considered opposite if intersection of sets that hold opposite assumptions is not empty. For such guard contexts, it can be statically proved that nested context will not be reachable under any possible configuration. There are two pairs of opposite assumptions: exists / not exists and feedthrough / not feedthrough .

Function effect inheritance

Inheritance of reachable effects is performed using following rules:

  1. If the effect is related to terminal access:
  2. If the terminal is guarded using feedthrough or not feedthrough in the context in which the function is called, it is mapped to the guarded equivalent of the effect and inherited
  3. Otherwise, the effect is directly inherited
  4. If the effect is ACCESS_UNGUARDED_OPTIONAL or DERIVE_UNGUARDED_OPTIONAL :
  5. If parameter variable is guarded using not exists in current context, an error is raised
  6. If parameter variable is guarded using exists in current context, the effect is discarded
  7. Otherwise, the effect is directly inherited
  8. Otherwise, the effect is directly inherited
  9. Guard context of inherited effect is union of original effect's guard context and the current guard context

Note

Only reachable effects are considered for inheritance, so it is sufficient to check if parameter variables of terminal effects are guarded in the current context using feedthrough or not feedthrough . As unreachable effects are discarded, only terminal effects for unguarded feedthrough access remain.

AST traversal

Effect analysis is performed as mostly left-to-right depth-first AST traversal. Guard context originating from guard blocks is stored in symbol table and is easy to track. Current type analysis supports only storing scopes per statement block, and does not support scopes per single statement or per single expression. Because of this, separate structure is needed to store inline guarded context. Ordinary order of AST traversal is broken when traversing elvis expressions. Right branch is traversed before left branch, because inline guarded not exists context can only be registered in preorder traversal of elvis expression, due to language grammar not having separate rules nodes for elvis expression branches. To prevent double traversal (as tree gets revisited in regualar traversal), sets of visited variables and function calls are maintained.

Top-level function effects

After analyzing function effects, a check is performed for top-level effects.

Top-level functions are functions that are not called by other functions. TML defines special built-in functions (such as output_fnc ). Only special built-in functions are allowed to have effects related to terminal access.

As top-level functions are dead code, and allowed terminal effects of custom functions cen be determined only in call context, determining allowed effects is not possible. A warning is raised that errors related to terminal access are ignored.

Global scope

Subset of the AST that consists of nodes that are not function definitions represents the global scope. Simple expressions with forbidden effects immediately raise an error. Function calls with forbidden effects raise an error.
