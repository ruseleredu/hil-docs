    Data-flow equations

    Preparation phase

    In preparation phase we do an AST walk in order to determine each variable definition (assignment) and usage (read) location. For this, we construct a form of symbol table where we have an entry for each variable in the program. Each entry holds additional meta-data required by the further analysis.

    Since TML is lexically scoped this symbol table will take a shape of a tree where the nodes are scopes. Thus, the root of the tree is the global scope. During the AST walk, whenever we cross the beginning of the syntax block which can have local variables we create a new node in the symbol table tree.

    For each variable assignment we record the location (e.g. a reference to assignment node) in the set of definitions for the given variable (a meta-data kept in the entry of the symbol table). This set of definition will be used as a KILL set in the DFA (see bellow). For each variable usage (read) we record location in another set in the symbol table entry. This information can be used in other type of analysis (e.g. liveness).

    As in the examples below, the symbol table might be represented as a lookup table that maps symbolic names of variables to the corresponding entries, each containing two sets, one with definition locations and another with read (use) locations of the matching variable. While doing an AST walk, at the moment of encountering the node that (re)defines a variable (the variable is on the LHS), we save a reference to the corresponding entry of the symbol table in the node. Similarly, when encountering a node that reads (use) the same variable (the variable is on the RHS), we save a reference to the matching entry in the node. Those references allow the direct access to the corresponding entries of the symbol table while doing the latter analyses without need to be aware of lexical scopes.

    In this phase we shall also mark expressions that must be static. This analysis start from all expression that has a must be static constituents (currently strings). When an expression contains a string it is marked as must be static. All variables in the expression must be static also for the whole expression to remain static and thus all other expressions that contain one of these variables are must be static. This information propagates through expressions/variables using work-list algorithm. The must be static information is used after DFA for loop unroll and if statements pruning.

    DF execution

    For the start we shall do Reachable Definitions Analysis (RDA) which is a form of Forward May analysis where for each variable usage point we calculate a set of definitions (assignment) which may influence the given location.

    Or, to specify it more precise:

    A definition of variable v reaches program point p if there exists a path with no intervening assignment to v

    This is also called def-use information.

    For DFA: - In is calculated as union of Out of all predecessors. - KILL information is found in the symbol table for the symbol at LHS of the assignment. It is a set of references to assignment locations collected in preparation phase. - GEN is a just a reference to the statement of the current assignment. - Out is calculated as Out = GEN U (In \ KILL)

    Now, a standard DF worklist based algorithm is used until fixpoint is reached (a state in which new iteration does not change outputs). See [^grune2012], chapter 5 and [^foster2011] slide 26.

    The result are In sets for each statement in the program (program point) which specifies reachable definitions, i.e. assignments that can influence the given statement. This can be used to determine for each variable usage if it can be statically evaluated and what will be the value.

    [^grune2012]: Dick Grune, Kees Van Reeuwijk, Henri E Bal, Ceriel JH Jacobs, and Koen Langendoen. Modern compiler design. Springer Science & Business Media, 2012. [^foster2011]: Jeff Foster. Dataflow analysis. 2011. lecture notes.
