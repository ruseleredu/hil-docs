    TML v0.2 SketchBook

    Introduction

    As @igord proposed, this file was created to serve as a sketchbook with initial thoughts about the development of the TML v0.2. Before any major coding, the TML developers should discuss about ideas through the sketchbook to avoid hacking in Python characteristic for so far TML development process.

    v0.2

    The major novelty that TML v0.2 should bring is the comprehensive Data Flow Analysis (DFA). The plan is to enrich the TML compiler with Control Flow Graph that serves as a foundation for better understanding the dynamic behavior of TML programs. The chapter 5 of the book: "Modern Compiler Design" [^grune2012] serves as a holy book for TML v0.2 developers. They obliged to carefully study it with the goal to develop the stable data flow analyzer. Beside optimal analyzer they all strive to, the code quality should be the one the most important property.

    The data flow analyzer consists of two major parts: - AST Threading - Data-flow Equations

    Multislicing

    This section provides some ideas and explanations on the new multislicing operation.

    Multislicing is an extension to slicing. Where slicing a tensor returns a sub-tensor, multislicing returns a tensor of tensors.

    For example, let x be of type tensor<int, 3, 4> (i.e. a matrix).

    Multislice expression:

    x[:*, :]

    would be of type tensor<tensor<int, 4>, 3>. E.g. each element of the resulting tensor will be a row from x or to be more specific the result would be [x[0, :], x[1, :], x[2, :]].

    While:

    x[:, :*]

    would be of type tensor<tensor<int, 3>, 4>. E.g. each element of the resulting tensor will be a column from x or [x[:, 0], x[:, 1], x[:, 2], x[:, 3]]

    Multislice can operate over slices with given boundaries. For example, these are all valid expressions and the resulting types:

    • x[:2*, :] - tensor<tensor<int, 4>, 2> ([x[0, :], x[1, :]])
    • x[:, 1:3*] - tensor<tensor<int, 3>, 2> ([x[:, 1], x[:, 2]])
    • x[1:, :2*] - tensor<tensor<int, 2>, 2> ([x[1:, 0], x[1:, 1]])

    We could generalize multislicing to produce higher dimension outer tensor.

    For example, let x be of type tensor<int, 3, 4, 5>. Then:

    x[1:*, :, :*]

    Would be of type tensor<tensor<int, 4>, 2, 5> or we get a matrix of vectors:

    [
    x[1, :, 0], x[1, :, 1], x[1, :, 2], x[1, :, 3], x[1, :, 3] ;
    x[2, :, 0], x[2, :, 1], x[2, :, 2], x[2, :, 3], x[2, :, 3] ;
]

    A note of calling functions on tensors

    TML functions operate either element-wise or over all scalar elements. If the inner elements are of tensor type functions are applied recursivelly.

    For example:

    • sin(x) where x is of type tensor<float, 3, 4> would return the same type where on each tensor element sin operation is performed.
    • average(x) is of scalar type if inner type of x tensor is scalar. If inner type of x tensor is a tensor then the return will be a tensor of the same dimension as the outer tensor. For example:
    x: tensor<float, 3, 4>
average(x) gives float

x: tensor<tensor<float, 2, 2>, 5>
average(x) gives tensor<float, 5> where average is applied to each inner tensor

    A note on relational operators

    Relational operator is performed element-wise when applied to tensors. If the tensors are nested the operation is performed recursivelly. Thus, the return type would be the same dimension but with the bool final inner type.

    For example:

    Let x be of type tensor<float, 3, 4>:

    Then:

    x > 5

    produces a value of type tensor<bool, 3, 4> where each element greater than 5 will produce true value.

    all and any

    These are functions that operate over all elements of tensor and produce scalar if innter type is bool. If inner type is tensor it is applied recursively until reaching scalars of bool type. It is an error if these functions are applied to tensors whose final scalar type is not bool.

    For example:

    x: tensor<bool, 2, 3>

all(x) -> returns true if all elements are true
any(x) -> returns true if any element is true

x: tensor<tensor<bool, 2, 2>, 3>
all(x) -> returns tensor<bool, 3>

    Some examples of condition tests:

    if any(x > 10)    # if any element of x is greater than 10
    ...

if all(x < 0)    # if all elements of x are negative

    Combining multislicing and function calls

    Multislicing is a powerfull operator which provides great flexibility.

    Let x be of type tensor<float, 5, 4>.

    To ask "is there any row with sin value greater than 0.5 we can write:

    if any(all((sin(x) > 0.5)[:*, :])))

    Lets break down this expression:

    First we calculate sin for each element of tensor x. The result is of the same type as x. 

    sin(x)

    Then we compare the result to 0.5 and get tensor<bool, 5, 4> with true value for each element whose sin is greater than 0.5:

    sin(x) > 0.5

    Then we are apply multislice over rows (first dimension) thus getting tensor<tensor<float, 4>, 5> or a tensor of all rows:

    (sin(x) > 0.5)[:*, :]

    Now, we are applying all recursively (because we have tensor of tensors) to all inner tensors (rows) and we get a tensor of booleans - tensor<bool, 5>:

    all((sin(x) > 0.5)[:*, :]))

    At the and we apply any to tensor of booleans. This time any won't recurse as we have a tensor of scalars and we get a single boolean value:

    any(all((sin(x) > 0.5)[:*, :])))

    To ask the same question for columns we would just do a multislice over columns:

    if any(all((sin(x) > 0.5)[:, :*])))

    We can see that using slicing/multislicing, function call and relational operators rules provide a great flexibility and consistency. The rules work the same accross all functions and relational operators thus once learned can be applied in a consistent manner.

    Note

    any can be optimized to return early as soon as the first element which satisfied condition is found. But, this should not influence the language usage and should be left to compiler optimization.
