Schematic Editor integration

Note

All TML related changes in Schematic Editor can be found on tml-develop branch.

TML compiler is called like any other compiler in SE, via utilizing the subprocess module.

Model preparation

Before the TML compiler is called, some small preparations are being made during creation of the reduced model. Since we are calling TML just like any other external compiler, we need to store the results (generated C code) in a directory that can be later on utilized by the code generator. In our case, we are creating a temp directory in convert_core_model() and use it for the current compilation.

Calling the TML compiler

The function that communicates with the TML compiler is called call_tml and can be found here: t_sw/thcc/typhoon/tools/compiler/code_generation/generator/compile.py::371 . The call_tml() calls TML compiler and stores the result (compiled C code) into the previously mentioned temp directory.

The compile command

TML compiler comes with a compile command that compiles all the components from the model that have TML implementation. The compile command comes with the following arguments:

  • model_file_path - a path to the serialized Schematic model
  • components_impl_path - a dir that contains component implementations in TML
  • --output-dir (optional) - a dir where the generated C code will be stored

Once the command is executed, the compiler will first create an object representation of the model. After that, all TML implementations from the dir defined by components_impl_path are loaded. Finally, the compiler will iterate through the model and compile every components that has a TML implementation.

If the optional --output-dir argument is omitted, the compiler will not generate any files, but it will still perform a complete analysis and even call the code generator. However, if the argument is provided, for each component, the following files will be generated:

  • <component_fqn>.var - defines all variables that will be used in the C code,
  • <component_fqn>.init_fnc - generated C code for the init_fnc,
  • <component_fqn>.output_fnc - generated C code for the output_fnc,
  • <component_fqn>.update_fnc - generated C code for the update_fnc,
  • <component_fqn>.fnc - generated C code for all user-defined functions,
  • <component_fqn>.proto - defines prototypes for all functions from TML file, and
  • <component_fqn>.error - contains all errors and warnings for the corresponding component.

``admonish note For all components in the model with a TML implementation corresponding C code is generated during call_tml()`, before the C code generator in SE is called. 

### stout, stderr, status code

The TML compiler will print all information during the compilation into the console.
Status message (i.e. compilation started/finished) will be printed to the `sys.stdout`.
Warnings and errors are printed to the `sys.stderr`.

If the compilation is completed successfully, the compiler will return `0` as the status code.
If the compilation of any of the components fails, the compiler will return `-1` as the
status code.

Schematic editor checks the status code in `call_tml()` function. If the TML compiler
doesn't return `0`, the compilation of the whole model will be stopped.

## Integration with SE C code generator

When preparing data for templates, C code generator detects whether the code
for the corresponding component is already generated by TML. If it is, the
code will be included into the template data (check `prepare_component_data()`
in `generator/base.py`).

```admonish note
Data is sent into templates as a dict. Meaning, generated code is stored in the
dict with `tml_` prefix (e.g. init function will be stored as `tml_init_fnc`).

In templates, for each function we simply check if the corresponding key is defined in the template data. For example, 

{# from common_int2_handler_function.template #}
{% for c in components_per_exec_rate[i] -%}
{% if "tml_output_fnc" in c %}
    {{c.tml_output_fnc}}
{% else %}
    {# code that calls template instead #}
{% endfor %}

To support user-defined functions, common_HIL_functions.template is updated with the following: 

// TML functions

{% for c in component_list -%}
{% if "tml_fnc" in c%}
{{c.tml_fnc}}
{% endif %}
{% endfor %}

Challenges

During the implementation of TML, there were several workaround (hacks) in SE side that should be addressed properly.

Next sections contain examples where we needed to introduce hacks to help TML, or bugs that often occurred.

``admonish note All issues that require changes in the Schematic Editor are labeled with schematic-rework` label. 

### Dynamic terminals ([Trac 6887](https://trac.typhoon-hil.com/ticket/6887))

We use `dynamic` keyword to define whether a terminal is dynamic or not. However, there is no way to tell if a terminal is optional (either exists or not), or we can have multiple instances of the terminal (i.e. input terminals for Sum, Product,...).

A much more informative way would be to use cardinal numbers. A cardinality of a terminal would be defined as a pair of lower and upper bounds. In our case, we will most often see the following cardinalities defined:

- (0, 1) - optional terminal. It can either not exist, or maximum one instance can be instantiated,
- (0, *) - dynamic terminal. No restrictions in min and max number of instances,
- (1, *) - dynamic terminal where at least one terminal must be instantiated
- (1, N) - dynamic terminal where a number of terminals must be between 1 and N.


For example, optional terminals can be defined like this:
in { cardinality=(0, 1) kind = sp label = "" position = top center direction = in sp_type { supported = int, real default = real } } 
A biggest advantage of this approach is that the validation can be integrated directly into Schematic Editor, and manual code for such validations can be removed.

In addition, this feature is needed for TML project to implement guards properly.

```admonish note
At the moment, this is handled manually in `reduced_sch_model/export.py`, where
`CARDINAL_NUM_PER_CMP_TERM` dict defines a cardinal number for each dynamic terminal.

Namespace variables ( Trac 6886 )

Currently, namespace variables are defined inside pre_compile or post_resolve handlers, in an arbitrary way.

Problem that often occurs is that a namespace variable is declared within if-else clause, which can cause problems during code generation. The latest example of such problem was with discrete_state_space.cmp where a template tried to access variable that is never defined: 

A_all_zeros, B_all_zeros, C_all_zeros, D_all_zeros, dss_mat_csr = convert_dense_dss_to_csr(A, B, C, D)
A_mat, B_mat, C_mat, D_mat = dss_mat_csr
if A_all_zeros is False:
    A_values, A_indices, A_indptr, A_values_list, A_indices_list, A_indptr_list, a_values_rows, a_indices_rows, a_indptr_rows, a_rows = A_mat
if B_all_zeros is False:
    B_values, B_indices, B_indptr, B_values_list, B_indices_list, B_indptr_list, b_columns, b_values_rows, b_indices_rows, b_indptr_rows, b_rows = B_mat
if C_all_zeros is False:
    C_values, C_indices, C_indptr, C_values_list, C_indices_list, C_indptr_list, c_values_rows, c_indices_rows, c_indptr_rows, c_rows = C_mat
if D_all_zeros is False:
    D_values, D_indices, D_indptr, D_values_list, D_indices_list, D_indptr_list, d_values_rows, d_indices_rows, d_indptr_rows, d_rows = D_mat

As we can see, there are cases where some of the variables are defined only if certain condition is met. This, in conjunction with pretty complicated templates, makes debugging extra hard.

In addition, with the development of TML, it would be really beneficial to have as complete component description as possible because it will provide more precise type checking and more concise C code.

One way to provide better definition of namespace variables would be to expand .cmp file with a section similar to properties and terminals. For example: 

namespace_vars {
   A_values {
     type = real[],
     default_value = [1, 2, 3]  // optional, if omitted default_value will be None
   }

}

The migration to more formal way of defining namespace variables can be implemented gradually. At the beginning, we can allow both formal and informal namespace variable definition, then slowly deprecate the informal way.

Note

We need to pay extra attention to user libraries, and we will probably need to add option for defining namespace variables in the Mask Editor.

Schematic model (de)serialization

The modules for creating a reduced schematic model and (de)serialization are tightly coupled with the Schematic core (check export.py and serialize.py for example).

In order to keep TML as separate as possible from THCC toolchain, a simplifier version of reduced model is implemented and can be found in tml/thcc module.

However, a major drawback of this approach is that there is always a possibility of getting out of sync with the modules in Schematic Editor.
