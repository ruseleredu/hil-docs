Optional type

TML supports optionally present variables. Terminals can be optionally present depending on the model configuration. User defined variables can be optionally present only if their type is derived from an optionally present terminal.

TML makes a distinction between optionally present variables and values . All variables, whether optionally present or not, must be defined in symbol table. When accessing optionally present variables, their presence can be proved only using exists guard block.

On the other hand, values in expressions can be optionally present only if the variable was inline guarded. Functions can be called in different contexts, and treating unguarded optional variables differently in separate guard contexts would be both confusing and hard to implement. Because of this, all unguarded accesses assume that a variable is guarded using a guard block, or raise an error otherwise.

Optional type inference

TMLOptional is a wrapper type that contains base type that can be either a tensor, a scalar or TML_MISSING . TML_ERROR is never wrapped inside TMLOptional . TMLOptional type is created only when variables are guarded using inline guard operator ? .

In binary expressions, if both types are TMLOptional , base type is unwrapped and new base type is inferred using regular type inference. Outer type remains optional, as both subexpressions can optionally be present.

If only one type is TMLOptional , and the other is not, resulting type will not be optional, as expression value is defined in all possible cases. If optional value is missing, inferred type is equal to the present type. If optional value is present, it's base type is unwrapped and type is inferred using regular type inference.

Optional type assignability

TMLOptional is not assignable to any type, regardless of which type is wrapped inside. To use TMLOptional in assignments, final elvis expression is needed. Elvis expression always returns a non-optional type, either unwrapped from left branch, or directly from right branch.
