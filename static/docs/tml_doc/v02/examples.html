    Intraprocedural analysis (RDA only)

    Simple linear CFG

    The following snippet shows a simple function with three nodes (marked as N1-N3) forming linear CFG.

    fn foo():
    int x = 5  # N1
    int y = 3  # N2
    x = y + 2  # N3
end

    Preparation phase

    The first step is to walk the AST to build a one-level symbol table containing metadata about variables x and y. In this example, we consider only set of definitions of the variables and ignore set of read locations. The sets of definitions contain references to the nodes that define or assign to the corresponding variables. The symbol table content is the following

    {
  x: {N1, N3};
  y: {N2};
}

    In this step, we determine the KILL sets of all nodes. As described above, the GEN set of a node contains the reference to that node.

    Reaching Definition Analysis

    For each node, we calculate In and Out sets iteratively until reaching the fixed point.

    ## The first and only iteration
# N1
In(N1) = EMPTY
Out(N1) = {N1}

# N2
In(N2) = OUT(N1) = {N1}
Out(N2) = {N1, N2}

# N3
In(N3) = OUT(N2) = {N1, N2}
Out(N3) = {N2, N3}

    Static Values Calculation

    For each node (program point), we determine whether the variable modified by the node is static, undefined or dynamic. The node N1 defines variable x. Since its In is empty and the RHS of the definition uses only literals, x is marked as static. Similarly, the y is static too. The node N3 redefines the value of x by using y and literal. Using the In set of N3, we determine that only one definition of y (inside the node N2) influences the value of x, meaning the both y and x are static.

    The CFG with an if-statement

    Consider the following snippet with if statement. Note that N4 points both to the N5 and N9.

    fn foo():
    int x = 0       # N1
    int y = x + 1   # N2
    if x > y:       # N3 is condition, N4 is if_statement node
        x = x + 1   # N5
        int x = 2   # N6
        x = x + 5   # N7
        y = x + 2   # N8
    end             # N9 is join_if
    z = x + y       # N10
end

    Preparation phase

    The symbol table has two nested scopes. The outer scope corresponds the foo function, while the nested scope matches the if branch. The first assignment to x inside the branch modifies the x of the outer scope, while the later definitions/assignment modifies the x of the inner scope. Finally, the symbol table has the following shape:

    {
  x: {N1, N5};
  y: {N2, N8};
  z: {N10};
  {
    # The scope corresponds to the if-branch.
    x: {N6, N7}
  }
}

    Reaching Definition Analysis

    The N4 nas two successors: N5 and N9, so the process of calculating In and Out looks like this:

    # N1
In(N1) = EMPTY
Out(N1) = {N1}

# N2
In(N2) = Out(N1) = {N1}
Out(N2) = {N1, N2}

# KILL(N3) = GEN(N3) = EMPTY
In(N3) = Out(N2) = {N1, N2}
Out(N3) = In(N3) = {N1, N2}

# KILL(N4) = GEN(N4) = EMPTY
In(N4) = Out(N3) = {N1, N2}
Out(N4) = In(N4) = {N1, N2}

# N5
In(N5) = Out(N4) = {N1, N2}
Out(N5) = {N2, N5}

# N6 
In(N6) = Out(N5) = {N2, N5}
Out(N6) = {N2, N5, N6}

# N7
In(N7) = Out(N6) = {N2, N5, N6}
Out(N7) = {N2, N5, N7}

# N8 
In(N8) = Out(N7) = {N2, N5, N7}
Out(N8) = {N5, N7, N8}

# KILL(N9) = {N6, N7}, GEN(N9) = EMPTY
In(N9) = Out(N8) U Out(N4) = {N5, N7, N8} U {N1, N2} = {N1, N2, N5, N7, N8}
Out(N9) = {N1, N2, N5, N8}

# N10
In(N10) = Out(N9) = {N1, N2, N5, N8}
Out(N10) = {N1, N2, N5, N8, N10}

    The join_if node is responsible to KILL all variables local for the if branch scope.

    Static Values Calculation

    Considering the RHS of the first node, one concludes the x is static in N1. The same node influences the definition of y inside N2, so both variables are static at this program point. The static property of both variables are transferred to node N3 and N4. The assignment to x in node N5 depends on the value of x inside N1, meaning the x is static in N5. Trivially, one can notice that the x belonging to the inner scope is static in both nodes N6 and N7. Assignment to y inside the node N8 is influenced by the assignment to x of the inner scope in N7, meaning both variables are static inside this node.

    Two assignments to the x of the outer corresponding the N1 and N5 nodes respectively influences join_if node N9, meaning the x is dynamic. The same stands for the y. Node N10 defines x by using the dynamic variable x and y, causing the z to become dynamic.

    If the node N has exactly one predecessor and the N does not modify the variable var, then the var has the same property (undefined, static, or dynamic) as in the N's predecessor. This property was implicitly assumed while describing the Static Values Calculation for all examples.

    The CFG with an if-else-statement

    The following code snippets shows a function containing if-else statement.

    fn foo():
    int x = 0       # N1
    int y = x + 1   # N2
    if x > y:       # N3 is condition, N4 is if_statement node
        x += 1      # N5 is equivalent to `x = x + 1`
    else:
        x += 2      # N6 is equivalent to `x = x + 2`
        y = x + 2   # N7
    end             # N8 is join_if
    z = x + y       # N9
end

    Preparation phase

    The preparation phase produces the following symbol table: 

    {
  x: {N1, N5, N6};
  y: {N2, N7};
  z: {N9};
  {
    # No local variables in the if-branch scope.
  };
  {
    # No local variables in the else-branch scope.
  }
}

    Reaching Definition Analysis

    Reaching definitions analysis behaves as follows:

    # N1
In(N1) = EMPTY
Out(N1) = {N1}

# N2
In(N2) = Out(N1) = {N1}
Out(N2) = {N1, N2}

# KILL(N3) = GEN(N3) = EMPTY
In(N3) = Out(N2) = {N1, N2}
Out(N3) = {N1, N2}

# KILL(N4) = GEN(N4) = EMPTY
In(N4) = Out(N3) = {N1, N2}
Out(N4) = {N1, N2}

# N5
In(N5) = Out(N4) = {N1, N2}
Out(N5) = {N2, N5}

# N6
In(N6) = Out(N4) = {N1, N2}
Out(N6) = {N2, N6}

# N7
In(N7) = Out(N6) = {N2, N6}
Out(N7) = {N6, N7}

# KILL(N8) = {local variables for if-else statement} = EMPTY
# GEN(N8) = EMPTY
In(N8) = Out(N5) U Out(N7) = {N2, N5} U {N6, N7} = { N2, N5, N6, N7}
Out(N8) = { N2, N5, N6, N7}

# In(N9) = Out(N8) = { N2, N5, N6, N7}
# Out(N9) = { N2, N5, N6, N7, N9}

    The node N8 should have killed all variables local for both if-branch and else-branch scopes. Since there is 0 variable local for these scopes, the node kills 0 variables.

    Static Values Calculation

    The RHS of the N1 uses literals only, so the x is static. One assignment to x influences y definition in N1, so both x and y are static at this point. Since N3 and N4 do not modify any state, the static properties of the two variables are transferred. The value of x in the node N5 is influenced by the assignment to this variable in the node N1, so the x is static at this point. Similarly, the x and y hold the static property in nodes N6 and N7 respectively.

    Finally, two assignment to x (from nodes N5 and N6) and two assignments to y (from nodes N2 and N7) influences the value of z. Thus, at this point of analysis we cannot determine single values of x and y in the node N9 so we shall treat them as dynamic and mark z variable as dynamic at this point. However, as we shall see bellow, by if statement pruning, as the condition is static, we can calculate and prove that only a single assignment to x and y influence N9 and thus we can prove that z is static.

    The CFG with a while-statement

    Consider the following snippet with a while statement. The N4 node has two successors: N5 and N8. The latter node has only one predecessor N4.

    fn foo():
    int x = 0           # N1
    int cnt = 0         # N2
    while cnt < 10:     # N3 is the condition, N4 is the while_statement node
        x = x + cnt     # N5
        cnt += 1        # N6 is equivalent to cnt = cnt + 1
        int y = x       # N7 points back to the N3
    end                 # N8 is the end_while node
   int z = x * cnt      # N9
end

    Preparation phase

    The preparation phase outputs the following symbol table:

    {
  x: {N1, N5};
  cnt: {N2, N6};
  z: {N9};
  {
    # The scope of the while-statement body
    y: {N7};  
  }
}

    Reaching Definition Analysis

    During the first iteration, the data-flow equation solver is aware only about the Out set of the N2 node, but not of the N7's Out set. Thus, the iteration takes two iterations to converges.

    The first iteration of the RDA behaves as follows:

    # N1
In(N1) = EMPTY
Out(N1) = {N1}

# N2
In(N2) = Out(N1) = {N1}
Out(N2) = {N1, N2}

# KILL(N3) = GEN(N3) = EMPTY since the N3 is the condition node
# As stated abode, the Out(N7) is not evaluated at the moment,
# so use the EMPTY as the appropriate value.
In(N3) = Out(N2) U Out(N7) = {N1, N2} U EMPTY = {N1, N2}
Out(N3) = In(N3) = {N1, N2}

# The while statement node does not kill nor generate any variables,
# thus the KILL(N4) = GEN(N4) = EMPTY
In(N4) = Out(N3) = {N1, N2}
Out(N4) = In(N3) = {N1, N2}

# N5
In(N5) = Out(N4) = {N1, N2}
Out(N5) = {N2, N5}

# N6
In(N6) = Out(N5) = {N2, N5}
Out(N6) = {N5, N6}

# N7
In(N7) = Out(N6) = {N5, N6}
Out(N7) = {N5, N6, N7}

# N8 kills variables local for the while-statement scope
# KILL(N8) = {N7}
In(N8) = Out(N4) = {N1, N2}
Out(N8) = {N1, N2}

# N9
In(N9) = Out(N8) = {N1, N2}
Out(N9) = {N1, N2, N9}

    The second iteration proceeds as follows:

    # The input sets of the N1 and N2 remains the same across iterations,
# so their outputs are unchanged.

# In this iteration, the solver uses the Out(N7) evaluated 
# by the previous iteration.
In(N3) = Out(N2) U Out(N7) = {N1, N2} U {N5, N6, N7} = {N1, N2, N5, N6, N7}
Out(N3) = In(N3) = {N1, N2, N5, N6, N7}

# N4
In(N4) = Out(N3) = {N1, N2, N5, N6, N7}
Out(N4) = In(N4) = {N1, N2, N5, N6, N7}

# N5
In(N5) = Out(N4) = {N1, N2, N5, N6, N7}
Out(N5) = {N2, N5, N6, N7}

# N6
In(N6) = Out(N5) = {N2, N5, N6, N7}
Out(N6) = {N5, N6, N7}

# N7
In(N7) = Out(N6) = {N5, N6, N7}
Out(N7) = {N5, N6, N7}

# N8: Consider that KILL(N8) = {N7}
In(N8) = Out(N4) = {N1, N2, N5, N6, N7}
Out(N8) = {N1, N2, N5, N6}

# N9
In(N9) = Out(N8) = {N1, N2, N5, N6}
Out(N9) = {N1, N2, N5, N6, N9}

    Static Values Calculation

    Obviously, x and cnt are static in N1 and N2 points. In sets of N3 and N4 indicates the x and cnt are dynamic in these two points. Two assignments to cnt (N2 and N6) and two previous assignment to x (N1 and N5) influence the value of x in N5, meaning the x and cnt remain dynamic in this node. Similarly, the cnt is dynamic in N6. The value of y at N7 program point depends on only assignment to x in N5. Recursively, we visit the node N5 to determine the dynamic nature of x at N5, so we conclude the y is dynamic in N7. From the In set of the node N8, one concludes that both x and cnt are dynamic. At the end, two assignments to x (and cnt respectively) influence the value of z, meaning it is dynamic at this program point.

    The CFG with a for-statement

    Consider the following snippet with a for statement. The N4 node has two successors: N5 and N8. The latter node has only one predecessor N4.

    fn foo():
    int x = 1           # N1
    for i = 1:10:        # N2 is the range, N3 is the node that initialize the i,
                        # N4 is the for_statement node
        x = x + i       # N5
        i = 2           # N6
        int y = i + 1   # N7 points back to the N3
    end                 # N8 is the end_for node
    int z = x * 3       # N9
end

    Preparation phase

    For the purpose of the for-statements, the TML compiler produces a so-called middle scope nested within the function foo scope, that at the same time encloses the scope of the loop body. Thus, the preparation phase assemble the following symbol table:

    {
  x: {N1, N5};
  z: {N9};
  {
    # The for-statement scope that defines the iteration variable.
    i: {N2, N6};
    {
      # The lexical scope belonging to the loop body
      y: {N7};
    }
  }
}

    Reaching Definition Analysis

    Similarly to the example with the while-statement, the equation solver is not aware of all Out sets corresponding to the N3's dynamic successors, meaning the solution process needs two iterations to stabilize.

    The following calculation is done as part of the first iteration.

    # N1
In(N1) = EMPTY
Out(N1) = {N1}

# N2 does not kill, nor generate any variables, so KILL(N3) = GEN(N3) = EMPTY
In(N2) = Out(N1) = {N1}
Out(N2) = {N1}

# At this moment, the Out(N7) is uknown, so we use the EMPTY as
# an appropriate value.
# N3
In(N3) = Out(N2) U Out(N7) = {N1} U EMPTY = {N1}
Out(N3) = In(N3) = {N1, N2}

# The for_statement not does not produce, nor kills any variables,
# so KILL(N4) = GEN(N4) = EMPTY
In(N4) = Out(N3) = {N1, N2}
Out(N4) = In(N4) = {N1, N2}

# N5
In(N5) = Out(N4) = {N1, N2}
Out(N5) = {N2, N5}

# N6
In(N6) = Out(N5) = {N2, N5}
Out(N6) = {N5, N6}

# N7
In(N7) = Out(N6) = {N5, N6}
Out(N7) = {N5, N6, N7}

# N8 kills variables local for the for statement loop 
# (iteration variable + loop body scope)
# KILL(N8) = {N2, N6, N7}
In(N8) = Out(N4) = {N1, N2}
Out(N8) = {N1}

In(N9) = Out(N8) = {N1}
Out(N9) = {N1, N9}

    The second iteration proceeds as follows:

    # The In sets of N1 and N2remain unchaged, so as theirs Out set.

# Use the Out(N7) calculated in the previous iteration.
In(N3) = Out(N2) U Out(N7) = {N1} U {N5, N6, N7} = {N1, N5, N6, N7}
Out(N3) = {N1, N2, N5, N7}

# N4
In(N4) = Out(N3) = {N1, N2, N5, N7}
Out(N4) = In(N4) = {N1, N2, N5, N7}

# N5
In(N5) = Out(N4) = {N1, N2, N5, N7}
Out(N5) = {N2, N5, N7}

# N6
In(N6) = Out(N5) = {N2, N5, N7}
Out(N6) = {N5, N6, N7}

# N7
In(N7) = Out(N6) = {N5, N6, N7}
Out(N7) = {N5, N6, N7}

# N8: KILL(N8) = {N2, N6, N7}
In(N8) = Out(N4) = {N1, N2, N5, N7}
Out(N8) = {N1, N5}

# N9
In(N9) = Out(N8) = {N1, N5}
Out(N9) = {N1, N5, N9}

    Static Values Calculation

    It is obvious that x is static in both N1 and N2. Since the RHS of the N3 uses literals only, one concludes the i variable is static.

    Interprocedural analysis (RDA only)

    The following few examples covers typical cases that should bring more insights about interprocedural dataflow analysis.

    Interprocedural analysis of pure functions

    Preparation phase

    Consider the following example with pure functions only.

    fn foo():
    a = 3      # N1
    bar()      # replaced by the N2
    b = a + 1  # N5
end

fn bar():
    # N2 is the entry node
    a = 4  # N3
    # N4 is the exit node
end

    TML compilers copies the CFG of bar at the place of the corresponding call. Nodes are enumerated respecting the previous concatenation of for and bar CFGs. Nodes N2-N4 are actually copies of the original nodes created by the AST threading process. For the brevity, enter and exits nodes of foo are omitted.

    The symbol table contains scopes of both foo and bar functions

    {
  foo : {
    a: {N1};
    b: {N5};
  }
  bar : {
    a: {N3}; 
  }
}

    Reaching Definition Analysis

    Assuming that the program execution starts from the foo, the RDA behaves as follows:

    # N1
In(N1) = EMPTY
Out(N1) = {N1}

# N2
In(N2) = Out(N1) = {N1}
Out(N2) = {N1}

# N3
In(N3) = Out(N2) = {N1}
Out(N3) = {N1, N3}

# KILL(N4) = {N3}
In(N4) = Out(N3) = {N1, N3}
Out(N4) = {N1}

# N5
In(N5) = Out(N4) = {N1}
Out(N5) = {N1, N5}

    Static Value Analysis

    The N1 assigns the scalar value to a, meaning the a is static at this point. Since the bar does not see nor change the a belonging to the foo, and N5 defines b, the a remains static until foo finishes. The node N3 defines its own a variable, and the RHS of the definitions indicates the static nature of the variable. At the end, the static value of a influences the definition of b inside N5, so the b is static at this point.

    Preparation phase

    While executing the preparation phase, the TML compiler uses the information about aliasing explained above to produce the following symbol table:

    {
  a: {N1, N7};
  main: {
    b: {N3};
    c: {N11};
    tmp: {N4, N8}
  };
  bar: {
    var: {N6};
  };
}

    As one may notice, the temporary variable tmp aliasing bar's second parameter is introduced for display purposes. We still need to decide whether to explicitly introduce the variable, or to hide it somewhere behind the scene.

of the callers inside the copy of each return statement of a callee.
