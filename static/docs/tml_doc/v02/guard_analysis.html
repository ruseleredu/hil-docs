    Guard analysis

    Guard analysis is a pass that is used to deremine whether all guarded terminals were used inside guard blocks, either directly, or inside nested blocks. It is performed after effect analysis, because effect analysis is needed to determine which terminals were accessed by functions.

    Guard analysis is implemented as a combined preorder and postorder traversal of AST. Information about currently guarded and used variables is stored in a stack of guard contexts. Guard context consists of directly guarded variables (variables guarded by the innermost guard block), inhertied guarded variables (variables guarded by all outer guard blocks) and used variables (variables used in this scope and all nested scopes).

    Guard analysis is performed using following steps: 1. In preorder visit of guard blocks, a new guard context is added to the stack. Variables directly guarded in this context using exists, feedthrough and not feedthrough are added to directly guarded variables. If parent context exists, all direct and inherited guarded variables are added to inherited guarded variables of new context. 2. All child statements are traversed, and variables accessed directly in code are added to the context list of used variables. Variables used by functions are marked as used if they originate from blocks reachable blocks (i.e. not nested inside an opposite guard block). 3. In postorder visit of guard blocks, their guard context gets removed from the stack. If a parent context exists, it inherits all used variables inside the child context. This is necessary for properly detecting usage of guarded variables in nested blocks. A check is performed if all directly guarded variables (that are not inherited guarded, i.e. that are not guarded both in this and outer blocks) are in used list. Error reporting is postponed for inherited guarded variables until their direct guarded context is reached. Note that if not used in this child context, inherited guarded variables can be used inside another child context, so error reporing must always correspond to their direct guard block.
