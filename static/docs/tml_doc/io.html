    IO

    To enable interacting with the outside world, TML support IO. IO in TML is based on the concept of directed IO variables. IO statements are similar to ordinary variable reads and writes, but the direction is enforced. Unlike output terminals, which can be read if initialized, reading output IO variables is not allowed.

    IO variables are similar to channels in Go and streams in C++, but are not queued. Depending on the type of IO variable, IO operations are performed either at the exact moment of reaching IO statement, or at the beginning or the end of simulation cycle. Programmers are required to properly choose IO type suitable for their use case and to properly test components.

    Supported IO types

    TML currently supports only IO communication with FPGA solver. Two implementations exist:

    1. Basic IO - synchronous blocking IO supported by all HIL devices
    2. DMA IO - partially synchronous, non-blocking IO supported by newer HIL devices

    Basic IO

    In basic IO, every operation is performed on the level of single scalar value and is blocking. Every write is globally visible from the moment the statement was reached, and separate reads of the same variable in the same simulation step can return different values. This means that IO speed degrades proportionally to the number of scalar elements. Basic IO is the safest option to use as it can be used for ordinary value passing and synchronization. If used for synchronization, special attention must be paid as whole simulation cycle can get stuck.

    DMA IO

    DMA input variables are synchronized on the beginning of simulation cycle, while output variables are synchronized at the end. This means that reading the same input IO variable multiple times during a simulation cycle will not block, but will always return the same value. Writing to the same output variable will not block, but only the last write will be globally visible. For this reason, DMA variables cannot be used for synchronization, and programmers must pay attention on exact value visibility requirements.

    Choosing IO type

    TML compiler chooses the IO type for FPGA IO variables based on the following criteria:

    1. Whether the variable was annotated to support DMA or not
    2. Whether DMA IO is enabled in the model settings
    3. Whether the target HIL device supports DMA IO

    Users are advised to test IO components for both DMA and basic IO. If model behaves incorrectly with DMA enabled, users should first test it with DMA disabled. DMA can potentially degrade performance of models that have a relatively small number of IO operations. Proper manual performance testing is required to make the decision on the IO type.

    Declaring IO variables

    IO variables are declared using special syntax similar to ordinary variable declaration. IO variables can be declared only in the global scope. Declaration of IO variable contains direction, value type, IO type and address. Only numeric scalars and vectors are currently supported value types (i.e. bool and str are not supported). Address is specified by an arbitrary expression that can be statically evaluated of type int.

    Following example shows IO variable declaration in global scope:

    # Basic IO variables
in<int, 0x1234> a
in<tensor<int, 5>, 0x1234> b

out<int, 0x1234> c
out<tensor<int, 5>, 0x1234> d

# DMA IO variables
in<int, 0x1234, dma> e
in<tensor<int, 5>, 0x1234, dma> f

out<int, 0x1234, dma> g
out<tensor<int, 5>, 0x1234, dma> h

    Using IO variables

    IO variables can be used only from entry functions (either directly, or transitively through calls). The only exception is that IO variables cannot be read in init_fnc. Using IO variables in global scope is not allowed, neither directly nor through function calls. Arrow operator <- is used for reading and writing IO variables to differentiate IO operations from ordinary variable usage. Same syntax is used for IO operations regardless of the type of IO variable. Type checking and broadcasting rules are applied to IO variables the same way they are applied for ordinary variables.

    Following example shows basic usage (reads and writes) of IO variables

    # Variables must be declared first
in<int, 5> a
out<real, 10, dma> b

fn init_fnc():
    # Variables can be read and written to in every entry function

    # IO write uses <- instead of = used in ordinary assignments
    b <- 5

    # IO read must use <- operator as prefix instead of using plain variable name
    x = <-a
    y = 5 + <-a + 6
end

    IO variables can be accessed like ordinary tensors, with all broadcasting rules applied. To use tensor IO variable access, ordinary IO and tensor indexing syntax are combined. Lowest granularity of IO access is single scalar. IO variables do not support shorthand assignments.

    Following example shows advanced usage (reads and writes) of IO variables, including broadcasting and single element access

    in<tensor<int, 5>, 5> a
out<real, 10, dma> b

fn init_fnc():
    # Broadcasting is performed like in ordinary assignments
    a <- 5

    # Tensor IO variables can be accessed at the granularity of single scalar
    a[0] <- 5
    x = <-a[1] + 10

    # IAB is applied to scalar IO variables like in ordinary expressions.
    # Indexing of scalar IO variables is ignored.
    x = <-b[10] + <-b[10, 20]
    b[5] <- 1
end
